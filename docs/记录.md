简单记录 EasyYolo 从零开始，一步一步走来的故事。

## 2026/04/30

今天做了一次比较大的交互体验升级。

之前项目中到处是浏览器原生的 `alert()` 和 `confirm()` 弹窗，风格割裂、无法定制。这次把四个独立模态框合并成了一个通用的消息弹窗系统，根据类型自动切换蓝/橙/绿三种配色。另外新增了一个三选对话框，在数据集上传时如果检测到同名，可以选"覆盖上传"、"直接复用已有"或"取消"，不用再手动去 datasets 目录删文件了。

还修了一个队列训练的竞态 bug：当最后一个任务完成时，队列内部的轮询器和全局轮询器同时检测到完成状态，会弹出两次提示。通过在队列完成时更新状态守卫，阻止了全局轮询器的重复触发。

顺便把训练回调里的 ETA 计算和指标提取重构成了独立函数，代码读起来清爽了不少。

## 2026/04/25

早停机制正式上线。

之前训练总是跑满全部轮数，哪怕模型早就收敛了还是在空转。这次利用了 YOLO 自带的 patience 参数——当验证指标连续若干轮没有提升，训练就会自动停下来。关键是：要能区分"正常跑完"和"提前早停"。

后端在训练完成后会对比实际轮数和设定轮数，如果实际少于设定，就标记为早停完成，记录下早停的轮次。这些信息同时写入了数据库。前端详情页也会显示一个橙色的"早停完成"标签，而不是绿色的"正常完成"。

做早停的过程中还顺手改了推理模型的加载方式——从构造函数里自动加载改成按需懒加载。因为训练时推理模型和训练模型同时占着显存，显卡扛不住。

## 2026/04/22

队列训练系统是整个项目里最复杂的功能之一，今天把这个坑填上了。

核心思路很简单：允许用户一次性排好几个训练任务，上一个跑完自动跑下一个，中间不需要人守着。但实现起来细节很多——

首先是任务模型的设计。每个任务需要知道自己用哪个基础模型——可以是系统现有的，也可以是上一个任务产出的新模型。这样就形成了模型迭代链：yolo11n → 任务一的产物 → 任务二的产物 → ...

其次是进度监控。队列跑起来后，前端通过全局状态轮询器显示实时的训练指标和进度条，同时队列管理器内部也有自己的轮询（`waitForTaskCompletion`），检测到当前任务完成后自动流转到下一个。

还有状态持久化。用户排好任务后如果刷新了页面，或者中途关了浏览器，再打开时队列应该还在。所以用 localStorage 把整个任务队列序列化保存，恢复时自动还原。

这次一口气写了 1400 多行 JS，是项目里写得最累也最有成就感的一个功能。

## 2026/04/21

加了停止训练按钮。

训练跑起来之后，有时候参数调错了或者想换个数据集，只能等着它跑完或者直接杀进程。现在的方案是利用 Python 的 `threading.Event()` 作为信号——用户点停止按钮 → 后端设置 Event → 训练线程在下一个 batch 开始时检查到信号 → 干净地退出。退出后会自动清理临时文件、释放 GPU 显存、重新加载推理模型。

停止粒度做到了 batch 级别，也就是说点下停止后最多等一个 batch 就能响应，不会像以前那样要耗完一整个 epoch。

## 2026/04/14

修了两个训练监控方面的问题。

一是 ETA（预计剩余时间）不准。原因是时间基准变量没有在训练开始时正确初始化，导致第一轮显示的预计时间完全是乱的。修好后，ETA 根据每轮的实际耗时乘以剩余轮数来估算，显示格式也做了人性化处理（比如"2小时 15分钟"）。

二是最后一轮的评估指标没有入库。YOLO 的回调在训练过程中实时推送指标，但验证指标（mAP、Precision、Recall）在最后一轮可能还没更新就被训练结束覆盖了。解决方法是在训练完成后，用 Ultralytics 返回的最终结果字典进行回填——只补回缺失的验证指标，不覆盖过程中采集的损失值。这样入库的数据就完整了。

## 2026/03/18

今天集中解决了几个模型管理方面的体验问题。

模型介绍之前存不进数据库——参数传递路径上有个环节漏了，修好后介绍就能正常显示和保存了。还加了编辑介绍的模态框，不用再去侧边栏找入口。

重命名模型时会遇到一个问题：其他模型可能是在这个模型的基础上训练的。如果直接改名字，那些子模型详情页里的"基础模型"超链接就断了。解决方法是在重命名的同时，把推理历史、训练记录、图表目录里的相关引用全部同步更新。

侧边栏的模型列表项上也加了一个悬浮菜单按钮（三个点），鼠标移上去就能看到重命名、改介绍、删除三个快捷操作。

## 2026/03/17

训练功能的核心代码在这一天完成，是整个项目最大的单次提交，加了将近 1500 行新代码。

训练架构方面，考虑到训练一次可能跑几十分钟甚至几小时，不能让它阻塞 FastAPI 的主线程。所以训练被放在了一个独立的后台线程里执行，前端通过每 2 秒轮询一次进度接口来获取实时的 epoch、指标和预计剩余时间。

数据集上传的处理花了不少心思。用户上传的是一个 zip 压缩包，里面可能是直接的图片+标签+data.yaml，也可能嵌套了一层目录。后端解压后会递归搜索 data.yaml 的位置，以它所在的目录作为实际的数据集路径。

GPU 显存管理也是一个需要处理的问题。推理模型和训练模型不能同时加载在显卡上——24G 显存也扛不住。所以训练开始前先把推理模型卸掉，训练结束后再重新加载回来。

产物的分发流程是：best.pt → 拷贝到 trained 目录；各类评估图表（12 种）→ 拷贝到 trainchart 目录下以模型名命名的子文件夹；数据库新增一条训练记录；最后清理临时 runs 目录。

## 2026/03/10

批量识别功能上线。

技术方案采用了 SSE（Server-Sent Events）。后端用 async generator 逐张处理图片，每处理完一张就推送一个事件给前端：`{"current": 3, "total": 99, "original_url": "...", "result_url": "..."}`。前端用 `ReadableStream` 的 reader 逐块读取响应，实时更新进度条和中间结果缩略图。

这是一种"流式渲染"的感觉，不用等所有图片都处理完就能看到结果，用户体验比同步请求好很多。单次上限设为 99 张。
