## 📅 2026/04/30
- **通用消息弹窗系统 (Unified Modal System)**：
    - **统一弹窗**：将独立的 `warningModal`、`stopQueueModal`、`taskFailedModal`、`switchModeModal` 四个模态框合并为一个 `messageModal`，通过 CSS class 切换 `info`（蓝）、`warning`（橙）、`success`（绿）三种颜色变体，顶部渐变条、标题颜色、确认按钮颜色随类型联动。
    - **三种按钮布局**：新增 `showMessage()`（单确认按钮）、`showConfirm()`（取消+确认双按钮）、`showThreeChoice()`（是/否/取消三按钮）三个函数，彻底替换原生 `alert()` 和 `confirm()`，所有函数共享同一个 HTML 结构，通过 `style.display` 控制各按钮的显隐。
    - **样式增强**：新增 `#messageModalCancelBtn` 和 `#messageModalNoBtn` 样式（透明背景 + 灰色边框 + hover 浅色底），与统一的设计语言对齐。

- **数据集上传去重检查 (Dataset Dedup Check)**：
    - **新增 API**：`GET /datasets` 遍历 `DATASETS_DIR` 下所有子目录，通过 `os.walk` 查找 `data.yaml` 所在位置，返回 `{name, path}` 列表供前端判断同名数据集。
    - **三选交互**：`handleDatasetUpload` 和 `handleMultiTaskDatasetUpload` 在上传前先调用 `api.getDatasets()` 检测同名，"是"覆盖重新上传、"否"直接复用已有路径（跳过上传）、"取消"清空 input 放弃操作。多任务模式下通过 `multiTaskManager.updateTask()` 更新对应任务的 `datasetPath` 和 `datasetName`。

- **队列训练竞态修复 (Race Condition Fix)**：
    - **问题根因**：`waitForTaskCompletion` 内部通过 `setInterval` 每 2 秒轮询训练状态，同时全局 `startTrainingPoller` 也以 2 秒间隔轮询。队列最后一个任务完成后，`moveToNextTask()` 将 `isQueueRunning` 置为 `false`，但 `state.lastTrainingStatus` 仍为 `'training'`，导致 poller 在下一个 tick 触发单任务完成弹窗。
    - **修复方案**：队列继续执行下一任务时设置 `state.lastTrainingStatus = 'training'`，队列全部完成时设置 `state.lastTrainingStatus = 'success'` 阻止 poller 重复触发。

- **训练回调重构**：将 `_compute_eta()` 和 `_extract_metrics()` 从 `on_train_epoch_end` 回调中提取为独立函数，前者根据每轮耗时 × 剩余轮数估算 ETA，后者通过 `_fuzzy_get` 模糊匹配不同 YOLO 版本的指标 key 名。

- **端口范围扩展**：`frontend-js.py` 端口扫描范围从 10 个扩展到 20 个，降低多人协作时的端口冲突概率。

## 📅 2026/04/25
- **早停机制 (Early Stopping Mechanism)**：
    - **状态追踪**：`training_state` 新增 `early_stopped`（bool）和 `early_stop_epoch`（int）字段，训练初始化时重置为 `False`/`0`。
    - **检测逻辑**：`model.train()` 返回后，对比 `training_state["progress"]`（实际完成轮数）与 `training_state["total"]`（设定总轮数）。若 `0 < actual < target`，判定为 patience 触发早停，记录早停轮次并打印日志。
    - **回调拆分**：将原来的 `on_fit_epoch_end` 拆分为 `on_train_batch_start`（仅检查停止事件，粒度为每个 batch）和 `on_train_epoch_end`（检查停止事件 + 更新进度/指标/ETA），使停止响应延迟从整轮降低到单个 batch 级别。
    - **数据库记录**：`add_training_record()` 新增 `early_stopped`（0/1）和 `early_stop_epoch` 参数，SQLite 表同步增加对应字段。
    - **推理模型懒加载**：`YoloService.__init__` 不再在构造时立即加载模型，改为在 `_ensure_model_loaded` 中按需加载，避免训练期间双模型同时占用 GPU 显存导致 OOM。
    - **详情展示**：前端 `ui.js` 的 `renderModelDetails` 根据 `early_stopped` 字段显示不同状态标签（绿色"正常完成" vs 橙色"早停完成"），并展示早停时的具体轮数。

## 📅 2026/04/22
- **队列训练系统 (Queue Training System)**（+1436 行 JS）：
    - **多任务管理器**：`multiTaskManager` 对象管理任务队列，核心状态包括 `tasks[]`（任务数组）、`currentTaskIndex`（当前执行索引）、`isQueueRunning`（队列运行标志）、`queueStatus`（idle/running/stopped/error/completed）。
    - **任务模型**：每个任务包含 `modelSource`（`'existing'` 使用现有模型 / `'previous'` 承接上个任务输出）、`baseModel`（基础模型名）、`newModelName`（产出模型名）、`datasetPath`、`parameters`（深拷贝 trainParams 避免共享引用）、`status`（pending/running/completed/failed）。
    - **状态持久化**：`saveToStorage()` / `loadFromStorage()` 通过 `localStorage` 实现任务队列的跨会话持久化，页面刷新后自动恢复。
    - **队列流转**：`startQueue()` 验证所有任务 → `executeNextTaskInQueue()` 获取当前任务并调用 `api.startTraining()` → `waitForTaskCompletion()` 以 2s 间隔轮询进度 → `moveToNextTask()` 将当前任务标记 completed 并递增索引 → 循环执行或结束。
    - **模式切换**：单任务/多任务模式通过 `switchTrainingMode()` / `applyTrainingMode()` 切换 UI 显示，切换至单任务时弹出确认框清空队列。
    - **任务卡片渲染**：`renderTaskList()` 通过 `<template>` 克隆填充任务卡片，动态绑定模型来源选择、基础模型下拉、数据集上传、参数编辑、删除等事件。

## 📅 2026/04/21
- **停止训练功能 (Training Stop Mechanism)**：
    - **信号机制**：使用 `threading.Event()` 作为停止信号。`stop_training()` API 被调用时设置 `stop_training_event.set()`。
    - **回调检测**：训练线程通过 YOLO 的 `on_train_batch_start` 回调在每个 batch 开始时调用 `_stop_now()` 检查 Event 状态，若被设置则抛出 `RuntimeError("Training stopped by user")`，外层 except 块捕获并将 `training_state["status"]` 置为 `"stopped"`。
    - **前端交互**：训练中按钮文案变为"停止训练"，点击后调用 `showConfirm` 确认 → `api.stopTraining()` → 后端设事件 → 训练线程在下一 batch 响应停止。`updateTrainButton()` 根据 `training_state.status` 动态切换按钮行为和文案。
    - **资源清理**：训练停止/失败后自动清理临时 `runs/detect/{model_name}` 目录，释放 GPU 显存（`gc.collect()` + `torch.cuda.empty_cache()`），重新加载之前的推理模型。

## 📅 2026/04/14
- **训练参数监控修复 (Monitoring Lag Fix)**：
    - **问题**：`on_fit_epoch_end` 回调中使用的 `last_epoch_time` 在第一次调用时未初始化，导致 ETA 计算不准。
    - **重构**：将 ETA 计算逻辑从 `on_fit_epoch_end` 中提取，通过 `_compute_eta(current_epoch)` 函数实现：记录每轮开始时间 `last_epoch_time`，用 `当前时间 - 上一轮时间 = 单轮耗时`，再乘以剩余轮数得到预计剩余秒数，格式化为"X小时 X分钟"或"X分钟 X秒"。
    - **启动脚本**：新增 `启动Vue前端.bat` 和 `启动后端.bat` 一键启动脚本。

- **最后一轮训练数据记录 (Final Epoch Metrics)**：
    - **指标回填**：YOLO 训练回调中的验证指标（mAP/Precision/Recall）在单轮训练中可能为 0 或缺失，训练结束后通过 `results.results_dict`（Ultralytics 权威返回）进行回填覆盖，只覆盖为 0 的指标保留已有的损失值。
    - **兜底策略**：若 `results_dict` 不可用，则尝试从 `results.box` 属性获取（`map50`/`map`/`mp`/`mr`），确保最终入库的指标准确。
    - **数据库记录**：`add_training_record` 的 `final_metrics` 字段使用经过回填校正后的 JSON 数据。
